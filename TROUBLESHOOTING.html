결함 01 · 데이터 정제

👻 비표시 문자 혼입

웹 크롤링 데이터에 사람 눈에 표시되지 않는 특수 공백 문자가 포함. 시스템 내부적으로는 "iPhone 17" 과 "iPhone□17" 이 서로 다른 문자열로 처리되어 검색 실패.

기술 상세
  • 원인 문자: 사람 눈에 보이지 않는 특수 공백 문자 (NBSP). Apple 페이지가 모델명이 줄바꿈으로 분리되지 않도록 의도적으로 삽입.
  • 침투 경로: 데이터 크롤링 시 NBSP 를 일반 공백으로 변환하지 않고 그대로 추출 → 파일명과 본문에 잔존.
  • 매칭 실패 지점: 벡터 DB 의 모델명 라벨이 NBSP 박힌 문자열로 저장됨 → 사용자가 입력한 일반 공백 질의와 글자 단위로 불일치.
  • 해결 적용: 유니코드 정규화 (NFKC) 를 크롤링 단계와 저장 단계 모두에 적용. NBSP·전각/반각·합자가 한 표준형으로 통일됨.

결함 02 · AI 안정성

🎲 비결정적 검색 결과

동일 질의 3회 반복 시 매회 상이한 결과 반환. 소형 AI 가 매 호출마다 검색 조건을 다르게 생성하는 것이 원인.

기술 상세
  • 기존 동작: 검색 도구가 사용자 질의를 AI 에 전달 → "검색 조건을 JSON 형식으로 만들어라" 라고 요청 → AI 가 자유 형식으로 출력한 텍스트를 JSON 으로 해석.
  • 실패 원인: 자유 형식 출력은 AI 의 일관성에 의존. 소형 AI (Gemma 4) 는 매 호출마다 다른 JSON 구조·필터 누락 등 비결정적 동작.
  • 해결 원리: 출력 가능한 값 자체를 제약. 51개 모델명을 선택지로 정의 (Pydantic Literal) → AI 가 51개 중 하나만 생성하도록 강제 (구조화 출력 강제).
  • 인스턴스 격리: 구조화 출력 전용 AI 와 답변 생성용 AI 를 별도 인스턴스로 분리. 구조화 제약의 잔향이 답변에 전파되지 않도록.
  • 검증: 동일 질의 3회 호출 → 3회 모두 동일 결과. 비결정성 100% 제거.

결함 03 · 언어 모델 약점

🤷 한국어 의도 오인식

"17 프로 맥스" 질의에 대해 AI 가 "iPhone 15 Pro Max" 로 오인식. 학습 데이터의 "15 Pro Max" 빈도가 높아 해당 모델로 매핑된 현상.

기술 상세
  • 잔존 결함: 출력 공간을 51개로 제한해도 51개 안에서 잘못된 선택을 막지 못함.
  • 원인 1 — "해당 없음" 회피: 소형 AI 가 "해당 모델 없음" 선택을 회피. 51개 후보 중 하나를 강제로 고르는 경향.
  • 원인 2 — 숫자 인식 약점: AI 가 "17" 같은 숫자를 잘게 분해해서 인식. "Pro Max" 같이 학습 빈도 높은 단어로 매핑이 쏠림.
  • Layer 1 — 시스템 메시지 강화: "모델 번호가 질의에 없으면 추측 금지", "도메인 외 질의는 모델 없음" 등 명시 규칙 추가. 시스템 메시지가 필드 설명보다 우선순위 높음.
  • Layer 2 — 코드 후처리: AI 출력 모델명의 숫자가 질의에 실제로 등장하는지 정규식으로 검증. 부재 시 "해당 없음" 으로 강제. AI 출력과 무관한 코드 차원의 가드.

결함 04 · 검색 구조

⚖️ 비교 질의 단일 응답

비교 요청 시 AI 가 한 모델 정보만 응답. 검색 도구가 단일 모델만 추출하도록 설계되어 비교 질의에서도 한 모델만 검색.

기술 상세
  • 결함 1 — 검색 도구의 단일 추출 구조: 추출 가능한 모델을 "한 개 또는 없음" 으로 정의 → 비교 질의가 들어와도 한 모델만 선택하도록 강제됨.
  • 결함 2 — 라벨 전달 누락: 답변 생성 단계의 기본 설정이 검색된 문서의 본문만 사용 → 어느 본문이 어느 모델 것인지 라벨이 답변 AI 에 전달되지 않음.
  • 해결 1 — 다중 추출 허용: 추출 모델을 "여러 개 목록" 으로 변경. AI 가 0~N개 모델 추출 가능.
  • 해결 2 — 모델별 분리 검색: 한 번에 전체 검색 (필터 사용) 대신 모델별로 따로 검색 후 합치기. 단일 질의가 한 모델로 편향되는 것 방지, 균등 분배 보장.
  • 해결 3 — 라벨 명시: 답변 AI 가 보는 문서 앞에 모델명 라벨을 붙임 (예: "[iPhone 17 Pro Max] 카메라..."). 본문과 모델의 매핑이 명확해짐.

결함 05 · 아키텍처

🏗️ 챗봇 구조 재설계

이전 4개 결함 분석 결과 — 모든 처리가 단일 함수에 결합된 것이 근본 원인. 8개 단계로 분리하여 각 단계 독립 점검 가능 구조로 재설계.

기술 상세
  • Before: 분기 도구 + 단일 함수에 5단계 결합 (이전 대화 통합·모델 추출·검색·정렬·로깅). 추적 수단은 콘솔 출력만.
  • After: 그래프 기반 8 노드 — 분기 · 맥락 통합 · 모델 추출 · 검색 · 정렬 · 응답 생성 · 검증 · 일반 대화.
  • 상태 전달: 상태 객체가 노드 간 입출력으로 흐름. 모든 단계 가시화 (질의·분기 결과·재구성된 질의·모델 목록·검색 결과·응답·이슈).
  • 스트리밍: 그래프가 AI 응답을 한 글자씩 흘려 보냄. 노드 식별자로 필터링하여 사용자에게 노출되는 노드의 출력만 전송 (SSE).
  • 온도 분리: 맥락 통합용 AI (0.0) / 응답 생성용 AI (0.0) / 일반 대화용 (0.2) 를 별도 인스턴스로 사용. 단일 흐름 구조에선 불가능했던 단계별 제어.
  • 확장 패턴: 검증 실패 시 응답 생성 재시도 (조건부 분기), 상태 영속화 (체크포인터), 모호 질의 명확화 노드 등이 노드 추가로 자연스럽게 도입 가능.
단일 함수 구조의 문제: 5개 단계 ①~⑤ 가 한 함수에 결합되어 장애 단계 추적 곤란. AI 응답 회피 감지 불가.
분리 구조의 이점: 각 노드 독립 점검 가능. 노드 클릭 시 단계별 역할 확인. "처리 흐름 시각화" 로 처리 순서 추적 가능.

구현 성과와 도출한 결론

01

외부 데이터는 진입 시점에서 정규화

웹 크롤링 데이터에는 비표시 문자가 혼입 가능. 진입 boundary 에서 정규화 적용 시 이후 모든 검색 안정성 확보.

02

AI 출력 제어는 권유보다 강제가 효과적

prompt 지시만으로는 빈번하게 무시. 출력 형식을 코드 레벨에서 제약하고 결과 후처리 검증으로 결정적 가드 확보.

03

검색 도구의 cardinality 가 응답 품질의 상한

비교 질의 응답을 위해 검색 도구의 다중 결과 추출 지원 필요. 도구 구조 결함 시 AI 성능과 무관하게 응답 생성 불가.

04

다중 처리를 단일 함수에 통합 회피

다단계를 단일 함수에 통합 시 장애 단계 추적 곤란. 명시적 단계 분리로 디버깅·확장·테스트 용이성 확보.

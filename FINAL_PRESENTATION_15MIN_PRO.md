## 🎤 발표 스크립트

### 1. 서론: 정보 과잉의 시대, 새로운 인사이트의 발견 (0:00 ~ 2:00)
**[Slide 1: 타이틀 및 팀 소개]**
안녕하세요. 데이터의 가치를 기술로 증명하는 팀9입니다. 저희가 준비한 프로젝트는 **"SummaryHub: AI 기반 YouTube 리뷰 요약 및 RAG 챗봇 플랫폼"**입니다.

**[Slide 2: 문제 정의 - 영상은 길고 시간은 부족하다]**
여러분, YouTube는 현대의 백과사전이지만, 동시에 '시간 도둑'이기도 합니다. 15분짜리 리뷰 영상 속에서 내가 정작 필요한 정보는 단 1분 내외인 경우가 많습니다. 하지만 우리는 그 1분을 찾기 위해 15분 전체를 소모하거나, 혹은 정보를 포기하곤 합니다.
저희는 이 '시간의 비대칭성'을 해결하고 싶었습니다. 사용자가 영상을 다 보지 않고도 핵심을 파악하고, 수집된 전문 데이터를 바탕으로 AI와 즉각적으로 소통할 수 있는 지능형 플랫폼을 구축한 배경입니다.

### 2. 아키텍처: Microservices Architecture와 I/O 격리 전략 (2:00 ~ 5:00)
**[Slide 3: 전체 시스템 아키텍처 다이어그램]**
저희 시스템의 가장 큰 특징은 **Microservices Architecture(MSA)**를 통한 '역할의 분리'입니다. 
Frontend, Backend-Core, Backend-Worker라는 세 가지 독립적인 레이어를 구축했습니다. 여기서 기술적으로 가장 고민했던 지점은 "왜 굳이 백엔드를 두 개로 나누었는가?"였습니다.

**[Slide 4: 왜 Node.js 워커인가? - I/O Blocking 해결]**
AI 요약은 구글 Gemini API를 활용하지만, 요청 후 응답까지 수십 초가 걸리는 **'Long-running I/O'** 작업입니다. 만약 메인 서버인 Django가 이 응답을 직접 기다린다면, 서버의 워커 쓰레드가 점유되어 다른 모든 사용자의 요청이 멈추는 병목 현상이 발생합니다.
저희는 이 부하를 완전히 격리하기 위해 Node.js 워커를 별도로 두었습니다. 비동기 이벤트 루프에 최적화된 Node.js가 AI와의 통신 및 자막 추출을 전담하게 함으로써, 메인 서버는 어떠한 상황에서도 쾌적한 응답 속도를 유지하는 '시스템의 가용성'을 확보할 수 있었습니다.

### 3. 핵심 기술 1: AI 자동 요약 파이프라인 (5:00 ~ 8:00)
**[Slide 5: 데이터 가공 흐름도]**
단순히 API를 호출하는 것만으로는 좋은 요약본을 얻을 수 없습니다. 저희는 정교한 파이프라인을 구축했습니다. 
사용자가 URL을 입력하면 워커가 `youtube-caption-extractor`로 자막을 가져오고, 이를 AI가 이해하기 쉬운 형태로 정제합니다. 

**[Slide 6: 프롬프트 엔지니어링과 마크다운 렌더링]**
여기에 저희만의 '프롬프트 엔지니어링'을 더했습니다. "리뷰어의 주관적 평가"와 "객관적 스펙 정보"를 구분하도록 지시하여 정보의 밀도를 높였고, 결과물은 마크다운 형식으로 생성됩니다. 
프론트엔드에서는 이를 실시간으로 렌더링하여 사용자에게 '잘 정리된 한 페이지의 리포트'를 보는 듯한 경험을 제공합니다. 이는 단순한 텍스트 나열보다 정보 습득 속도를 3배 이상 향상시켰습니다.

### 4. 핵심 기술 2: RAG 시스템과 데이터 품질의 사투 (8:00 ~ 11:00)
**[Slide 7: RAG 시스템의 원리와 ChromaDB]**
단순 요약의 한계를 넘기 위해 도입한 것이 **RAG(검색 증강 생성)**입니다. AI의 고질적인 문제인 '환각 현상'을 잡기 위해, 저희는 **ChromaDB**라는 벡터 데이터베이스를 구축하고 검증된 데이터를 미리 학습시켰습니다.

**[Slide 8: 트러블슈팅 - API 결측치 문제와 크롤링]**
이 과정에서 큰 난관이 있었습니다. 처음에는 기존 제품 정보 API를 사용하려 했으나, 데이터의 **결측치**가 너무 많아 챗봇의 신뢰도가 떨어졌습니다. 
저희는 과감히 API 사용을 포기하고, Selenium과 BeautifulSoup을 이용한 **직접 크롤링 파이프라인**으로 선회했습니다. 제조사 공식 사이트에서 직접 데이터를 추출하고 정제함으로써 결측치 없는 고품질 지식 베이스를 완성했습니다. 
이제 챗봇은 AI의 기억이 아니라, 저희가 발로 뛰어 수집한 '팩트 데이터'를 보고 답변합니다. 이것이 저희 챗봇이 100% 신뢰할 수 있는 정보를 내놓는 비결입니다.

### 5. 설계의 정수: 휴먼 에러를 최소화하는 전략 (11:00 ~ 13:00)
**[Slide 9: 백엔드 안정성 설계 - CBV와 Nested Routers]**
시스템의 규모가 커질수록 개발자의 실수는 치명적입니다. 저희는 '휴먼 에러'를 줄이기 위한 설계를 강화했습니다. 
반복적인 로직은 **클래스 기반 뷰(CBV)**를 사용하여 코드량을 줄였고, `Nested Routers`를 도입해 복잡한 URL 구조를 자동화함으로써 경로 설정 오류를 원천 차단했습니다.

**[Slide 10: 보안 및 프론트엔드 아키텍처 - Token 기반 인증]**
보안 역시 직접 구현의 위험을 피하고 검증된 `dj-rest-auth`를 사용했습니다. 특히 **Token 기반의 Stateless 인증**을 적용하여 MSA 환경에서의 서비스 간 확장을 용이하게 했습니다. 
프론트엔드에서는 **Vue 3와 Pinia**를 연동해 사용자 인증 상태를 중앙에서 관리하며, Apple 스타일의 미니멀 디자인 시스템을 통해 기술적 복잡함을 우아한 사용자 경험으로 승화시켰습니다.

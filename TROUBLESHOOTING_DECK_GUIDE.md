## 0. 출발점이 된 지시사항

이 deck 는 다음 사용자 피드백을 누적하며 형태가 잡혔다. 새 프로젝트에서 deck 를 만들 때 **동일한 원칙** 으로 검토할 것.

1. **"마크다운을 그대로 옮기지 말 것"** — 트러블슈팅 `.md` 의 구조를 그대로 옮기면 deck 가 안 된다. 문제 발견 → 해결법 → 결과 순서로 **재구성**.
2. **"포트폴리오에 쓸 수 있게 PPT 같은 형식"** — 발표 가능한 슬라이드, 1슬라이드 = 1메시지.
3. **"비전문가도 이해 가능하게"** — 도메인 지식 없이도 따라올 수 있게 평어체 요약을 우선, 기술 용어는 명시적으로 보조 박스에 격리.
4. **"인터랙티브 요소가 반드시 있어야 함"** — 정적 텍스트만 있는 슬라이드는 deck 의 의도에 맞지 않음. 클릭/토글/실행 버튼 중 하나는 반드시.
5. **"다이어그램은 마크다운 그대로 가져오지 말고 SVG/HTML 로 다시 그릴 것"** — ASCII art 나 코드 블록 다이어그램 금지. 인터랙티브하게 다시 그린다.
6. **"하룻동안 했다는 식의 시간 메타데이터는 불필요"** — "X일 작업", "Y시간 디버깅" 같은 부수 정보는 빼고 결함·해결·결과만.
7. **"비율은 16:9"** — 발표용으로 16:9 고정, 반응형 포기.
8. **다이어그램은 "가독성·강조 시 가독성 손실" 까지 검증** — 글씨 크기, 종료 노드의 시각적 차별화, 흐름 강조 애니메이션에서 글씨가 안 보이는 경우 등 모두 점검 대상.
9. **콘텐츠는 핵심에 집중** — "작은 모델 선택" 같이 deck 의 메인 메시지와 무관한 분기 내용은 슬라이드에서 통째로 제거하고 결과 페이지까지 재작성.

## 1. 컨셉

### 1.1 deck 의 목적

- **문서가 아니라 발표용 deck**. `.md` 를 슬라이드로 그대로 옮기지 않고, **스토리텔링 흐름** 으로 재구성한다.
- 청중은 **두 부류** 를 동시 대응: 비전문가(맥락만 알고 싶은 사람) + 기술자(세부 구현이 궁금한 사람).
- 한 슬라이드 안에서 **좌측 = 평어체 설명 + 인터랙티브 데모**, **우측 박스 = 기술 상세 bullet** 으로 분리. 발표자가 어느 쪽을 가리킬지 선택 가능.

### 1.2 1:1:1 매핑 원칙

> **1 슬라이드 = 1 결함 = 1 인터랙티브 데모**

- 텍스트만 있는 슬라이드 금지 (Hero / Overview / 결론 슬라이드 제외).
- "이 결함이 한 클릭으로 보일 수 있는 차이가 있는가?" 에 답하지 못하면 슬라이드로 만들지 않는다.
- "수정 전 / 수정 후" 토글이 결함 슬라이드의 **기본 컴포넌트**.

### 1.3 슬라이드 구성 (총 9~10 장)

```
1. Hero          — 무엇을 만들었는가 (1줄)
2. 결과물 데모   — 실제 동작 (가짜 데이터라도 클릭 가능)
3. 결함 overview — N개 카드, 클릭 시 점프
4..N-1. 결함     — 한 결함당 한 슬라이드, 인터랙티브 데모 포함
N. 결과·교훈·로드맵
```

## 2. 기술 결정 사항

그대로 가져갈 수 있는 선택들.

| 항목 | 선택 | 이유 |
|---|---|---|
| 무대 크기 | `1280×720` 고정 + JS `scale()` | 16:9 보존 (반응형 포기) |
| 슬라이드 전환 | `display: none/flex` + `fadeIn` keyframe | 모든 슬라이드 DOM 유지, active 클래스만 토글 |
| 키보드 | `← → Space PageUp/Down Home End 1-9 F D` | `INPUT/TEXTAREA` 포커스 시엔 무시 |
| 풀스크린 | `requestFullscreen()` | `F` 키 + 버튼 |
| 테마 | `html.dark` 클래스 + CSS 변수 + localStorage | 변수 한 줄로 다크모드 전체 적용 |
| 폰트 | Pretendard / Noto Sans KR + JetBrains Mono | macOS·Windows 공통 깔끔 |
| 색상 변수 | `--primary/danger/success/info/accent + --p-color` | 슬라이드별 강조색 차등 |

| 외부 의존성 | **없음** — 단일 HTML, JS/CSS 인라인 | 어디서 열어도 동작, 오프라인 발표 가능 |

### 2.2 16:9 fit 핵심 코드

```js
function fitDeck() {
  const w = window.innerWidth;
  const h = window.innerHeight;
  const scale = Math.min(w / 1280, h / 720);
  document.getElementById('deck').style.transform =
    `translate(-50%, -50%) scale(${scale})`;
}
window.addEventListener('resize', fitDeck);
fitDeck();
```

## 3. 인터랙션 패턴 카탈로그

deck 의 모든 결함 슬라이드는 아래 5개 패턴 중 하나에 매핑된다. **결함 도메인(데이터/AI/네트워크 등) 과 무관하게 재사용 가능**. 새 프로젝트의 결함이 어느 패턴에 해당하는지 먼저 분류하라.

### P1. Before/After 토글 — 같은 입력의 다른 결과

> 동일한 입력 데이터에 대해 수정 전·후 상태를 한 클릭으로 비교.

- **컨트롤**: 2개 버튼 (`.mode-toggle` / `.compare-toggle`) 로 모드 전환.
- **결과 영역**: 같은 위치에 다른 내용 렌더링.
- **검증 배너**: 결과가 옳은지 여부를 `.verdict-banner.fail` / `.ok` 로 색상 즉시 표시.
- **언제 쓰나**: 입력→출력 1:1 관계가 명확한 결함. 잘못된 응답 vs 올바른 응답.
- **예시**: 동일 질의에 대한 응답이 결함 적용 전엔 1개만, 후엔 N개로 나오는 비교.

### P2. 반복 실행 — 일관성·결정성 시각화

> 같은 입력을 N회 실행하고 결과를 나란히 보여줘서 변동성을 가시화.

- **컨트롤**: "🔁 N회 실행" 버튼 + Before/After 토글.
- **결과 영역**: N×M 그리드의 점·박스·배지. 매 실행마다 점이 채워지는 애니메이션.
- **데이터**: `Math.random()` 또는 사전 정의된 시퀀스로 가짜 비결정성 시뮬레이션.
- **언제 쓰나**: 결함이 "매번 다른 결과" 형태일 때. 안정성·결정성·재현성 이슈 전반.
- **예시**: 3회 반복 검색의 결과 점 5개씩 (수정 전 = 색깔 섞임, 수정 후 = 모두 동일).

### P3. 케이스 선택 — 분기별 거동 차이

> 여러 케이스를 카드로 늘어놓고, 사용자가 클릭하면 결과 영역이 그 케이스의 거동을 보여준다.

- **컨트롤**: 2~4개 카드 (`.q-card`) + 검증 ON/OFF 토글.
- **결과 영역**: 단일 응답 박스, 카드 선택에 따라 내용·색상 변경 (`.answer-box.empty/wrong/right`).
- **언제 쓰나**: 결함이 "특정 케이스에서만 발생" 일 때. 경계 조건·에지 케이스·도메인 외 입력.
- **예시**: 모호한 질의 / 도메인 외 질의 / 정상 질의 / 인사말 4개 카드 → 각 케이스의 응답 시연.

### P4. 입력 가시화 — 보이지 않던 것을 보이게

> 사람 눈으로는 동일하나 시스템 내부에서는 다른 입력을 가시화하는 토글.

- **컨트롤**: "👁 사람 눈" / "🤖 시스템 내부" 토글 + "🔍 실행" 버튼.
- **결과 영역**: 동일 텍스트가 모드에 따라 다르게 렌더링 (e.g. 빨간 마커, 점, 색상).
- **언제 쓰나**: 데이터·인코딩·표현 차이가 원인인 결함. 일반 사용자에게는 "왜 이게 문제인지" 가 보이지 않는 경우.
- **예시**: 비표시 문자가 사람 눈엔 공백이지만 시스템에선 다른 코드포인트로 인식되는 케이스.

### P5. 구조 다이어그램 + 단계 강조 — 흐름·아키텍처 변화

> SVG 노드 그래프 + Before/After 토글 + "▶ 흐름 시각화" 버튼으로 처리 순서 가시화.

- **컨트롤**: Before/After 토글 + 흐름 재생 버튼 + 노드 hover/click 정보 패널.
- **결과 영역**: SVG 노드들이 순차적으로 `.flowing` 클래스를 받아 색상 변경, 엣지에 `stroke-dasharray` 애니메이션.
- **언제 쓰나**: 결함이 "구조 자체" 인 경우. 모놀리식 → 모듈식, 동기 → 비동기, 단일 함수 → 파이프라인.
- **주의**: 노드 정보 패널은 **4가지 닫기 방식 모두 구현** (X 버튼 / 같은 노드 재클릭 / 다른 노드 클릭 / 외부 클릭) — 안 그러면 "닫을 수 없는" 케이스 발생.

### 패턴 선택 기준 (요약)

| 결함의 본질 | 패턴 |
|---|---|
| 입력→출력 단일 비교 | P1 |
| 반복 시 변동성 | P2 |
| 케이스별 거동 차이 | P3 |
| 입력 자체가 안 보임 | P4 |
| 처리 구조 변경 | P5 |

**가이드라인**: 결함이 어느 패턴에도 매칭되지 않으면, 그 결함은 deck 화하지 말고 텍스트 문서에 남겨라. 모든 결함을 deck 에 넣을 필요는 없다.

## 4. 공통 슬라이드 구조

모든 결함 슬라이드는 동일한 좌·우 레이아웃을 따른다.

```html
<section class="slide" style="--p-color: #f97316">  <!-- 결함별 강조색 -->
  <div class="problem-slide">
    <div class="problem-left">
      <div class="problem-num">결함 0X · 카테고리</div>
      <h2>🎯 결함 제목</h2>
      <p class="summary">
        1~2 문장 평어체 요약. <strong>핵심 키워드 강조.</strong>
      </p>
      <div class="problem-tech">
        <h5>기술 상세</h5>
        <ul>
          <li><strong>원인</strong>: ...</li>
          <li><strong>해결</strong>: ...</li>
          <li><strong>검증</strong>: ...</li>
          <!-- 4~5개 bullet -->
        </ul>
      </div>
    </div>
    <div class="interact">
      <div class="interact-label">⚙ 인터렉티브 데모</div>
      <h3>한 줄 데모 제목</h3>
      <!-- 패턴 P1~P5 중 하나의 컨트롤 + 결과 영역 -->
    </div>
  </div>
</section>
```

좌측 비율(`.problem-left`) : 우측 비율(`.interact`) 의 grid 비율은 보통 `1fr 1.4fr`. SVG 다이어그램이 들어가면 `1fr 1.5fr` 로 우측 확장.

## 5. 텍스트 규칙

deck 의 톤·구조 일관성을 위한 규칙.

- **각 결함당 4~5개 bullet** — "기존 동작 / 원인 / 해결 원리 / 검증 / 적용 위치" 같은 표준 슬롯.
- **이모지를 메타데이터처럼 사용** (장식 아닌 기능):
  - 결함 아이콘: 한 결함에 하나 고정 (👻 🎲 🤷 ⚖️ 🏗️ 같이 도메인 무관 픽토그램).
  - 상태: ✅ ❌ ⏳ 🔍 🛠 (검증 결과·진행 상태).
  - 카테고리 라벨: ⚙ 📝 💬 📚 (섹션 헤더).
- **bullet 안 `<strong>` 라벨링**: "**원인**: ...", "**해결**: ..." 발표 중 빠르게 스캔 가능.
- **"수정 전 / 수정 후" 라벨 통일** — 모든 슬라이드에서 동일한 텍스트 사용.
- **결과 배너 2종 고정**: `.verdict-banner.fail` (빨강) / `.ok` (초록). 검증 결과는 항상 색으로 즉시 인식.
- **시간/일자 메타데이터 제외** — "X일 작업", "Y시간 디버깅" 같은 부수 정보는 deck 에 넣지 않는다 (사용자 명시 지시).
- **한국어 평어체 우선**, 기술 용어는 인라인 코드(`monospace`) 로 마킹. 발표 시 자연스럽게 읽을 수 있어야 함.

## 6. JS 아키텍처

- 슬라이드별 JS 를 **블록 주석으로 명확히 분리**:
  ```js
  // ========== Slide 5 — Pattern P4 (NBSP) ==========
  // ========== Slide 6 — Pattern P2 (Determinism) ==========
  ```
- **모든 데모는 하드코딩된 가짜 데이터** 로 동작. API 호출 없음. 배포 단순 + 오프라인 발표 가능.
- 전역 상태는 슬라이드별 `let xxxMode = 'before'` 패턴 + 슬라이드 간 격리.
- `await sleep(ms)` 헬퍼 하나로 타이핑·반복실행·흐름강조 애니메이션 모두 처리.
- `addEventListener` 중복 등록 금지 (한 번씩만).
- SVG 노드는 `<g class="svg-node">` 단위로 묶고, 텍스트엔 `pointer-events: none` 으로 클릭 가로채기 방지.

## 7. 새 프로젝트 적용 체크리스트

1. **결함 목록 추리기** — 3~5개가 적당 (overview 카드가 한 행에 깔끔). 트러블슈팅 문서를 다 옮기지 말고 임팩트 있는 것만.
2. **각 결함을 §3 패턴에 매핑** — P1~P5 중 어느 것? 매칭 안 되면 슬라이드로 만들지 않는다.
3. **deck 메인 메시지에 무관한 분기 내용 제거** — 슬라이드만 빼는 게 아니라 결론·로드맵 페이지도 동시에 갱신해야 일관성 유지 (사용자 명시 지시).
4. **색상 변수 한 줄로 톤 조정** — `--primary` 만 바꾸면 deck 색 통일.
5. **결함 슬라이드부터 작성**. Hero / 결과물 / 결론은 마지막에 — 톤이 잡힌 뒤 시작·끝을 맞춰야 일관성 올라감.
6. **기존 HTML 을 템플릿으로 복사** → 결함 슬라이드의 좌측 텍스트 + 우측 인터랙티브 데모 블록만 새 결함으로 교체. 슬라이드 1, 3(overview), 마지막(교훈)은 텍스트만 갈아끼움.
7. **다이어그램 가독성 점검** (3대 항목):
   - 글자 크기가 16:9 무대에서 충분한가
   - 시작/종료 노드가 시각적으로 차별화돼 있는가 (모양·색)
   - 흐름 강조 애니메이션 시 색이 입혀져도 글자가 보이는가 (대비)
8. **Chrome 풀스크린 + 키보드 테스트**. `F` `←` `→` `1-9` `D` 모두 동작 확인.
9. **`@media print` 로 PDF export 시도** — 슬라이드 1장 = PDF 1쪽.

## 8. 함정 — 시행착오로 학습한 것

새 프로젝트에서도 반복될 가능성이 높은 실제 함정들.

### 8.1 다이어그램 가독성 (사용자 명시 지적 사례)

- **글씨 크기**: 16:9 무대에 SVG 가 들어가면 viewBox 좌표계 안에서 12~13px 폰트로 시작하면 거의 안 보임. **14~15px 부터 시작**.
- **종료 노드 차별화**: 모양·색·테두리 두께·crown 효과 등으로 일반 노드와 시각적 분리 필수. 같은 사각형은 종료가 아닌 중간 단계로 오해 발생.
- **강조 시 가독성**: 노드에 색이 입혀지는 흐름 시각화에서, 어두운 색을 채우면 검은 글자가 사라지는 케이스 발생. **흐름 색 입혔을 때 자동으로 글자 색을 `white` 로 토글** 하는 CSS 규칙 필요:
  ```css
  .svg-node.flowing .node-text { fill: white; }
  .svg-node.flowing .node-sub { fill: rgba(255,255,255,0.92); }
  ```
- **노드 겹침**: SVG viewBox 가로폭이 컬럼 수에 비해 좁으면 노드가 겹친다. 사용자 지시 사례 — "END 노드가 다른 노드와 겹치고 있음 → **전면적인 노드 재배치** 고려". 새 노드 추가 시 viewBox 크기부터 재검토.

### 8.2 슬라이드 컨테이너

- **`.slide` 에 `overflow: hidden` 필수**. 안 그러면 큰 SVG 가 다음 슬라이드로 삐져나옴.
- **다크모드 누락 스타일**: `.msg.bot`, `.typing` 같은 흰 배경 컴포넌트는 `html.dark` 분기에서 `var(--slide-bg)` 로 명시 재정의. 누락 시 다크모드에서 흰 박스가 둥둥 떠 있음.
- **이모지 + 텍스트 베이스라인**: 결과 점(`.result-dot`) 같이 이모지·텍스트를 한 박스 안에 같이 넣을 때 폰트 차이로 베이스라인 어긋남. `line-height: 1` + `display: flex; align-items: center` 명시 권장.

### 8.3 사용자 인터랙션

- **노드 정보 패널 닫기 4가지 모두 구현**:
  - (a) X 버튼
  - (b) 같은 노드 재클릭
  - (c) 다른 노드 클릭
  - (d) 슬라이드 외부 클릭
  하나라도 빠지면 "닫을 수 없는" 케이스.
- **키보드 단축키와 입력창 충돌**: 슬라이드 내부에 `<input>` 이 있을 경우 `1-9` 점프가 입력값과 충돌. `keydown` 핸들러 진입부에서 `['INPUT','TEXTAREA'].includes(e.target.tagName)` 체크 필수.

### 8.4 콘텐츠

- **모바일 대응은 의도적으로 포기**. 1280×720 무대를 작은 화면에서 보면 그대로 축소돼 글자 안 보임. 발표용이므로 OK. 사전 합의 필요.
- **슬라이드 순서도 의도적 설계**. 작은 결함 → 큰 결함 → 통합 아키텍처 결론 순. Before/After 가 가장 임팩트 있는 결함을 끝 쪽에.
- **메인 메시지에 무관한 분기는 슬라이드에서 빼고 결론도 같이 갱신**. 슬라이드만 빼면 마지막 결론·로드맵 페이지가 어색해진다 (사용자 명시 지시 사례).

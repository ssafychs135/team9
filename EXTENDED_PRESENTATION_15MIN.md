## Part 2. 발표 스크립트 (Speech Script)

### 1. 서론: 정보 과잉의 시대, 새로운 솔루션 (0:00 ~ 2:00)
안녕하세요. 저희는 YouTube 영상 데이터의 가치를 극대화하는 플랫폼, SummaryHub를 개발한 팀9입니다.

오늘날 우리는 정보의 홍수 속에 살고 있습니다. YouTube는 가장 강력한 정보원이지만, 동시에 가장 많은 시간을 소모하게 만드는 매체이기도 합니다. 15분짜리 리뷰 영상에서 내가 원하는 '단 1분의 정보'를 찾기 위해 우리는 너무 많은 시간을 씁니다.

저희는 이 문제를 두 가지 기술적 축으로 해결하고자 했습니다. 
첫째, **AI를 이용한 초정밀 영상 요약**입니다. 
둘째, 요약된 데이터를 넘어 검증된 지식 베이스를 바탕으로 소통하는 **RAG 기반의 지능형 챗봇**입니다. 

단순히 편리한 도구를 만드는 것을 넘어, 대규모 언어 모델(LLM)의 한계인 '정보의 부정확성'을 엔지니어링으로 어떻게 극복했는지 지금부터 상세히 설명드리겠습니다.

### 2. 아키텍처: Microservices Architecture (2:00 ~ 5:00)
저희 플랫폼의 전체 구조를 보시면, 세 개의 독립적인 서비스가 유기적으로 연결된 **MSA(마이크로서비스 아키텍처)**를 채택하고 있음을 알 수 있습니다.

여기서 가장 중요한 기술적 결정은 **"왜 백엔드 워커를 별도로 분리했는가?"**입니다. 
AI 요약 작업은 구글 Gemini API를 활용하지만, 요청을 보내고 응답을 받기까지 상당히 긴 **대기 시간(Long-running I/O)**이 발생합니다. 만약 메인 서버인 Django가 이 긴 응답을 직접 기다리게 된다면, 서버의 워커 쓰레드가 점유되어 다른 사용자의 요청을 처리하지 못하는 병목 현상이 발생하게 됩니다.

저희는 이러한 **비정형 데이터의 가공과 긴 대기 시간이 필요한 네트워크 I/O 부하**를 메인 서버로부터 완전히 격리시키기 위해 Node.js 기반의 워커를 구축했습니다. Node.js의 비동기 이벤트 루프는 이러한 대기 시간이 긴 작업들을 효율적으로 처리하는 데 매우 최적화되어 있습니다. 

결과적으로 Django 서버는 인증과 데이터 관리에 집중하며 쾌적한 응답 속도를 유지하고, 무거운 AI 작업은 전용 워커가 비동기적으로 수행하는 **'역할의 분리'**를 통해 시스템 전체의 안정성과 동시 처리 능력을 극대화했습니다.

### 3. 기술 심층 1: AI 자동 요약 파이프라인 (5:00 ~ 8:00)
이제 구체적인 기능을 살펴보겠습니다. 첫 번째 핵심 기술인 **AI 자동 요약**입니다.

이 과정은 단순한 API 호출이 아닙니다. 정교한 데이터 파이프라인이 존재합니다. 
사용자가 URL을 입력하면 워커 서비스가 `youtube-caption-extractor`를 통해 자막을 추출합니다. 하지만 자막 데이터는 오타가 많고 타임스탬프가 섞여 있어 그대로 AI에게 줄 수 없습니다. 

저희는 이 데이터를 정제한 뒤, **Google Gemini AI** 모델에 최적화된 프롬프트를 주입합니다. 단순히 "요약해줘"가 아니라, "리뷰어의 주관적인 의견과 객관적인 스펙 정보를 분리하여 마크다운 형식으로 작성하라"는 구체적인 지침을 부여합니다. 
결과물은 `marked.js`를 통해 프론트엔드에서 미려한 문서로 렌더링됩니다. 사용자는 영상을 보지 않고도, 텍스트만으로 영상의 모든 인사이트를 얻을 수 있습니다.

### 4. 기술 심층 2: RAG 시스템과 데이터 품질의 사투 (8:00 ~ 11:00)
두 번째이자 가장 핵심적인 기술은 **RAG(Retrieval-Augmented Generation)** 시스템입니다.

이 시스템을 구축하며 저희가 마주한 가장 큰 도전은 바로 **'데이터의 품질'**이었습니다. 
당초 저희는 기존에 공개된 제품 정보 API를 활용하여 데이터를 수집할 계획이었습니다. 하지만 실제 데이터를 분석해보니 심각한 문제가 있었습니다. 주요 사양 정보에 **결측치(Missing Values)**가 너무 많아, AI가 참조할 데이터로서의 가치가 현저히 떨어졌던 것입니다. 

'쓰레기가 들어가면 쓰레기가 나온다(Garbage In, Garbage Out)'는 원칙 아래, 저희는 과감한 결단을 내렸습니다. 편리한 API 사용을 포기하고, **직접 크롤링 파이프라인을 구축**하기로 한 것입니다. 
Selenium과 BeautifulSoup을 사용하여 제조사 공식 웹사이트와 전문 리뷰 사이트에서 데이터를 직접 추출했고, 이 과정에서 정교한 전처리 과정을 거쳐 결측치 없는 고품질의 지식 베이스를 완성할 수 있었습니다. 

이렇게 확보한 정밀 데이터를 **ChromaDB**라는 벡터 데이터베이스에 저장했습니다. 사용자가 질문을 던지면 시스템은 다음과 같이 동작합니다. 
1. **유사도 검색**: 질문의 의미를 분석해 직접 수집한 고품질 '팩트 데이터'를 찾습니다. 
2. **컨텍스트 주입**: 찾아낸 정밀 팩트를 LLM에게 참고 자료로 제공합니다. 
3. **근거 기반 답변**: AI는 자신의 기억이 아닌, 저희가 발로 뛰며 수집한 '검색 결과'를 바탕으로 답변을 구성합니다. 

이러한 데이터에 대한 집요함 덕분에, 저희 챗봇은 최신 제품 정보에 대해서도 '환각 현상' 없이 100% 신뢰할 수 있는 답변을 내놓을 수 있게 되었습니다.

### 5. 설계의 정수: 휴먼 에러를 최소화하는 전략 (11:00 ~ 13:00)
백엔드 아키텍처 설계 시 저희가 가장 집착했던 키워드는 바로 **'안정성'**과 **'휴먼 에러의 최소화'**입니다. 개발자가 작성하는 코드가 많아질수록 버그가 생길 확률도 높아지기 때문입니다.

이를 위해 저희는 세 가지 핵심 전략을 사용했습니다.
첫째, **Class-Based View와 ViewSet의 적극적인 활용**입니다. 반복되는 CRUD 로직을 직접 코딩하지 않고 DRF의 ViewSet을 상속받아 구현했습니다. 이를 통해 불필요한 보일러플레이트 코드를 줄이고, 비즈니스 로직에만 집중하여 코드의 가독성과 안정성을 동시에 확보했습니다.

둘째, **Nested Routers를 통한 API 규격화**입니다. 리뷰와 댓글처럼 계층 구조가 명확한 리소스를 다룰 때, 수동으로 URL 경로를 설정하면 경로 누락이나 잘못된 매핑이 발생하기 쉽습니다. 저희는 `drf-nested-routers`를 도입하여 상속 구조에 따른 URL을 자동 생성함으로써, 경로 설계 단계에서의 휴먼 에러를 원천 차단했습니다.

셋째, **Token 기반 인증 시스템과 dj-rest-auth의 도입**입니다. 인증은 가장 실수하기 쉽고 그 결과가 치명적인 분야입니다. 저희는 직접 인증 로직을 구현하는 위험을 감수하는 대신, 검증된 `dj-rest-auth`를 도입하여 **Token 기반의 Stateless 인증**을 구현했습니다. 세션 기반 인증과 달리 서버에 상태를 저장하지 않는 토큰 방식은, 저희 프로젝트와 같은 MSA 환경에서 서비스 간 확장을 매우 용이하게 만듭니다. 또한, 프론트엔드와 백엔드가 분리된 구조에서 API 보안을 유지하기 위한 최적의 선택이었습니다. 이는 보안 표준을 엄격히 준수하면서도 개발 효율성을 극대화하는 전략적 선택이었습니다.

결국 기술적 선택의 기준은 "얼마나 많은 코드를 짜느냐"가 아니라, **"얼마나 신뢰할 수 있는 시스템을 구축하느냐"**에 있었고, 이러한 도구들의 도입이 그 해답이 되었습니다.

### 6. 결론 및 향후 계획 (13:00 ~ 15:00)
프로젝트를 진행하며 가장 큰 도전은 서로 다른 언어와 프레임워크를 사용하는 서비스 간의 데이터를 맞추는 일이었습니다. 특히 벡터 DB의 임베딩 성능을 최적화하는 과정에서 수많은 시행착오를 겪었지만, 이를 통해 진정한 의미의 Full-stack 역량과 AI 엔지니어링 감각을 익힐 수 있었습니다.

향후 저희 플랫폼은 다음과 같은 확장을 계획하고 있습니다. 
첫째, **Docker 컨테이너화**를 통한 클라우드 배포 최적화입니다. 
둘째, 영상의 음성뿐만 아니라 화면의 텍스트와 이미지를 동시에 분석하는 **멀티모달 AI 요약** 기능을 추가할 예정입니다.
